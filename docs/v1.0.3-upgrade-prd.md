## 1. 업그레이드 배경

### 1.1 핵심 문제

**사용자 피드백**:
> "Novel Writer는 처음 30장까지 효과가 좋았지만, 30장 이후부터 AI가 초점을 잃고 이전 설정과 규칙을 잊기 시작했습니다."

**근본 원인**:
- 장편 소설 컨텍스트 누적으로 AI 주의력 분산
- 강제 컨텍스트 리로드 메커니즘 부재
- 체계적 지식 라이브러리 지원 부족
- 캐릭터/플롯 추적이 수동 기억에 의존

### 1.2 대상 사용자

- **주요 사용자**: 전문 작가, 50만 자 이상 장편 소설 창작
- **보조 사용자**: 초보 작가, 체계적 창작 가이드 필요
- **향후 계획**: VSCode 플러그인 개발을 위한 아키텍처 준비

## 2. 솔루션 설계

### 2.1 핵심 아키텍처 업그레이드

#### 지식 라이브러리 시스템 (Knowledge Base)

**설계 철학**: 계층형 아키텍처, 필요 시 로딩

```
.claude/knowledge-base/
├── README.md              # 키워드 매핑 테이블 + 총 인덱스
├── genres/                # 장르 지식 라이브러리
│   ├── romance.md        # 로맨스 (948행, ~27KB)
│   ├── mystery.md        # 미스터리 (1123행, ~31KB)
│   ├── historical.md     # 역사 (1127행, ~27KB)
│   ├── revenge.md        # 복수 (846행, ~22KB)
│   └── wuxia.md          # 무협 (143행, ~2.8KB)
├── craft/                 # 글쓰기 기법 (예약)
└── references/            # 참고 자료 (예약)
```

**토큰 효율**:
- 기존 방식: 50개 Skills × 40 tokens = ~2000 tokens
- 새로운 방식: 1개 탐지기(100 tokens) + 필요 시 지식 라이브러리(500 tokens) = 600 tokens
- **75% 토큰 비용 절감**

**키워드 자동 활성화 메커니즘**:
```yaml
romance:
  keywords: [로맨스, 사랑, 연애, 낭만, 감정선, 관계 아크, CP, 달달함, 애틋함]
  auto_load: genres/romance.md

mystery:
  keywords: [미스터리, 추리, 탐정, 수사, 수수께끼, 단서, 진실, 범인, 범죄]
  auto_load: genres/mystery.md
```

#### 품질 보증 Skills

**4가지 핵심 Skills**:

1. **setting-detector** (566행)
   - 기능: 사용자 입력의 키워드를 자동 탐지하여 해당 지식 라이브러리 로딩
   - 활성화 시점: 사용자가 스토리 장르, 설정, 배경을 언급할 때
   - 예시:
     ```
     사용자: "1920년대 로맨스 복수 소설을 쓰고 싶어요"
     탐지기 인식:
       - "1920년대" → china-1920s.md
       - "로맨스" → romance.md
       - "복수" → revenge.md
     3개 지식 라이브러리 자동 로딩
     ```

2. **pre-write-checklist** (541행)
   - 기능: `/write` 실행 전 AI가 9개 핵심 파일을 로딩했는지 강제 확인
   - 해결 문제: **30장 이후 초점 상실 문제의 핵심 해결책**
   - 강제 체크리스트:
     ```markdown
     ✓ 1. memory/constitution.md - 창작 헌법
     ✓ 2. memory/style-reference.md - 스타일 참조
     ✓ 3. stories/*/specification.md - 스토리 규격
     ✓ 4. stories/*/creative-plan.md - 창작 계획
     ✓ 5. stories/*/tasks.md - 현재 작업
     ✓ 6. spec/tracking/character-state.json - 캐릭터 상태
     ✓ 7. spec/tracking/relationships.json - 관계 네트워크
     ✓ 8. spec/tracking/plot-tracker.json - 플롯 추적
     ✓ 9. spec/tracking/validation-rules.json - 검증 규칙
     ```

3. **forgotten-elements** (147행)
   - 기능: 캐릭터/플롯/복선의 등장 빈도를 백그라운드 모니터링
   - 임계값:
     - 캐릭터: 10장 미등장 → 알림
     - 플롯 라인: 12장 진전 없음 → 알림
     - 복선: 20장 미회수 → 알림
   - 알림 예시:
     ```
     ⚠️ 캐릭터 알림:
     "조연 이명"이 10장 동안 미등장 (마지막: 제5장)
     - 다시 등장시킬 필요가 있나요?
     - 이 캐릭터의 이야기가 끝난 건가요?
     ```

4. **getting-started** (224행)
   - 기능: 신규 사용자 안내, 7단계 방법론 소개
   - 활성화 조건: 사용자가 "소설을 쓰고 싶어요", "어떻게 사용하나요" 등 말할 때
   - 내용:
     - 7단계 프로세스 상세 설명
     - 자주 묻는 질문 답변
     - 첫 사용 권장 사항

## 4. 지식 라이브러리 내용 상세

### 4.1 로맨스 소설 지식 라이브러리 (romance.md)

**핵심 내용** (948행):

1. **관계 아크 설계**
   - 첫 만남 → 호감 → 탐색 → 확인 → 승온 → 위기 → 화해 → 안정
   - 각 단계의 대표적 이벤트
   - 리듬 제어 기법

2. **감정 비트**
   - 설렘 순간 설계 (15+ 클래식 장면)
   - 달달한 순간 공식
   - 애틋한 플롯의 균형 조절

3. **케미스트리 연출**
   - 대화 은근한 끌림 기법
   - 보디랭귀지 세부사항
   - 내면 독백 설계

4. **갈등 관리**
   - 오해의 합리성 검사
   - 삼각관계 캐릭터의 올바른 활용
   - 외부 압력 vs 내부 모순

### 4.2 미스터리 추리 지식 라이브러리 (mystery.md)

**핵심 내용** (1123행):

1. **페어 플레이 원칙 (Fair Play)**
   - 모든 단서를 독자에게 제시해야 함
   - 핵심 정보를 숨기면 안 됨
   - 진실은 추론 가능해야 함

2. **단서 배치**
   - 3층 구조: 명시적 단서 + 암묵적 단서 + 레드 헤링
   - 단서 밀도: 5장마다 최소 3개의 유효 단서
   - 역추적 검증: 진실 공개 후 자체 모순이 없어야 함

3. **용의자 관리**
   - 3-5명 용의자 풀
   - 각자 동기 + 기회 + 의심 포인트 보유
   - 제거 순서의 리듬 설계

4. **진실 공개**
   - 도미노식 공개 vs 일괄 공개
   - 반전의 조건
   - 복선 회수 체크리스트

### 4.3 역사 소설 지식 라이브러리 (historical.md)

**핵심 내용** (1127행):

1. **대사건 vs 작은 디테일**
   - 역사적 대사건은 배경으로, 주선으로 삼지 말 것
   - 역사 속 인물의 선택에 집중
   - 흐름식 역사 서술 방지

2. **백묘 수법 (Historical White Description)**
   - 간결하고 힘있는 환경 묘사
   - 현대적 형용사 삭제
   - 예시 비교:
     ```
     ❌ 나쁨: "화려한 궁전이 금벽 휘황하여 감탄을 자아냈다"
     ✅ 좋음: "붉은 벽, 노란 기와, 처마 끝이 솟아올랐다"
     ```

3. **시대 분위기 연출**
   - 의복, 예절, 호칭의 정확성
   - 일상 생활 세부사항의 고증
   - 언어 스타일의 시대감

4. **역사 정확성 vs 오락성 균형**
   - 핵심 사건은 변조 불가
   - 세부사항은 합리적 허구 가능
   - 소인물 이야기의 공간이 넓음

### 4.4 복수 소재 지식 라이브러리 (revenge.md)

**핵심 내용** (846행):

1. **깊은 원한의 수립**
   - 원한에 충분한 이유가 있어야 함
   - 초반 상세한 복선 필요
   - 독자가 공감해야 함

2. **통쾌한 리듬**
   - 점진적 > 한 번에
   - 작은 것부터 큰 것으로, 점진적 업그레이드
   - 매번 새로운 방식

3. **성장 합리성**
   - 치트키가 너무 과하면 안 됨
   - 성장에는 대가가 필요
   - 무뇌 압도 방지

4. **도덕적 균형**
   - 복수에도 한계선이 있어야 함
   - 무차별 피해 금지
   - 정의감 vs 통쾌함

### 4.5 무협 소설 지식 라이브러리 (wuxia.md)

**핵심 내용** (143행, 간소화 버전):

1. **강호 규칙**
   - 의리 정신
   - 문파 체계
   - 은원 관계

2. **무공 체계**
   - 내공 vs 외공
   - 초식 묘사 기법
   - 실력 등급 설정

3. **협객 정신**
   - "대협은 나라와 백성을 위해"
   - 개인 원한 vs 대의
   - 통쾌한 원한 해소의 한도

### 5.1 글쓰기 프로세스 최적화

**이전 (v1.0.2)**:
```

  ↓
AI가 자체적으로 파일 읽기
  ↓
핵심 파일 누락 가능
  ↓
30장 이후 초점 상실
```

**현재 (v1.0.3)**:
```
사용자 /write 실행
  ↓

  ↓
AI가 읽은 9개 핵심 파일을 반드시 명시적으로 나열
  ↓
컨텍스트 완전성 확인 후 글쓰기 시작
  ↓
50장 이상 일관성 유지
```

### 5.2 지식 라이브러리 자동 활성화

**시나리오 예시**:

사용자 입력:
> "1920년대 상하이 배경의 복수 로맨스 소설을 쓰고 싶어요. 여주인공은 가수이고, 남주인공은 조직 보스입니다."

**setting-detector 자동 인식**:
- 키워드 "1920년대" + "상하이" → 역사적 배경
- 키워드 "복수" → revenge.md
- 키워드 "로맨스" → romance.md
- 키워드 "조직" → 특정 시대 지식 트리거 가능

**자동 로딩**:
- `knowledge-base/genres/historical.md` (~1100행)
- `knowledge-base/genres/revenge.md` (~850행)
- `knowledge-base/genres/romance.md` (~950행)

**총 토큰 소비**: 약 1500-1800 tokens

**가치**:
- AI가 즉시 3개 전문 지식 라이브러리 지원 확보
- 사용자 수동 지정 불필요
- 글쓰기 품질 현저히 향상

### 5.3 신규 사용자 가이드

**이전**:
- 사용자가 직접 7단계 방법론을 탐색해야 함
- 어디서 시작할지 모름
- 단계를 건너뛰거나 누락하기 쉬움

**현재**:
- "소설을 쓰고 싶어요" → getting-started 자동 활성화
- 완전한 프로세스 설명 제공

- 간단한 프로젝트부터 시작 권장

| 방식 | 토큰 소비 | 커버 범위 |

**새 지식 라이브러리 추가 비용**:
1. `knowledge-base/genres/`에 새 파일 생성
2. `README.md`에 키워드 매핑 추가
3. 코드 수정 불필요

**예시**: "SF" 지식 라이브러리 추가
```yaml
# knowledge-base/README.md 추가:
scifi:
  keywords: [SF, 공상과학, 우주, 미래, 로봇, AI, 사이버펑크]
  auto_load: genres/scifi.md
```
